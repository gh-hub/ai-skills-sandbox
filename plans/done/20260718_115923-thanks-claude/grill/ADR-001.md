# ADR-001: Monorepo architecture and API contract

## Status
Accepted

## Context
Need a minimal but complete monorepo for a "like Claude" feature: a static frontend, an API, and a persisted Postgres store, wired together for local Docker Compose use.

## Decision
- **Layout:** `apps/api` (NestJS), `apps/web` (Next.js static export), `packages/shared-types` (or similar) for the DTO/entity shapes shared by both apps.
- **Package manager / workspace linking:** pnpm workspaces.
- **Data access:** Drizzle ORM against Postgres, with drizzle-kit for migrations.
- **API contract:**
  - `POST /likes` — body: `{ story?: string; hoursSaved?: number }`, both optional, validated via `class-validator` DTOs (`CreateLikeDto`). Returns the created like.
  - `GET /likes/count` — returns `{ count: number }`, the total number of likes ever created.
  - Global `ValidationPipe` (whitelist + transform) applied at bootstrap.
  - `@nestjs/swagger` mounted at `/api/docs`, generated from the same DTOs.
- **Schema:** single `likes` table — `id` (uuid, pk), `created_at` (timestamptz, default now), `story` (text, nullable), `hours_saved` (numeric, nullable).
- **Frontend:** Next.js `output: 'export'`. Fetches `POST /api/likes` and `GET /api/likes/count` relative to its own origin.
- **Serving / networking:** docker-compose with three services — `postgres`, `api`, `web`. `web` is an nginx image serving the static export and reverse-proxying `/api/*` to `api:3000`, so the browser only ever talks to the `web` origin. No CORS configuration needed on the API.

## Consequences
- Adding a second frontend consumer later would need its own CORS handling (nginx proxy trick only covers the one static site).
- No dedup means the like count is a raw click count, not a unique-visitor count — acceptable per explicit requirement.
- Drizzle's migration files become the source of truth for schema changes; must run drizzle-kit generate/migrate as part of API startup or a documented manual step (to be decided at ticket/implementation time).
