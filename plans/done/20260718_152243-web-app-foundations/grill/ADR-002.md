# ADR-002: Tailwind CSS v4 + shadcn/ui as the styling stack

## Status
Accepted

## Context
`apps/web` has no CSS framework, no global stylesheet, and no component library — every element on the single existing page is unstyled native HTML. The app needs dark mode, and the user asked for shadcn/ui specifically.

## Decision
Install Tailwind CSS v4 and shadcn/ui together via shadcn's `init` CLI flow, which scaffolds `components.json`, Tailwind config/theme tokens, and the CSS variable-based light/dark theme structure shadcn/ui expects. Pull in components (Button, Input, Textarea, Form, Label, and a header/nav) as needed rather than all at once.

## Consequences
- Styling becomes utility-class-based (Tailwind) with shadcn/ui components as copy-in source (not an npm-installed component library) — component source lives in the repo (`apps/web/components/ui/`) and is directly editable.
- Dark mode theming (ADR/decision: next-themes) relies on shadcn/ui's CSS-variable convention (`.dark` class toggling), which shadcn's init sets up automatically — no separate dark-mode CSS architecture needed.
- Future components should be pulled from shadcn/ui's registry via its CLI rather than hand-rolled, to stay consistent with the established pattern.
- This is a greenfield adoption — no migration cost from a prior CSS approach, since none existed.
