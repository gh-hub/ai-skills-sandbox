# ADR-002: drizzle-kit as a production dependency, pruned Docker images

## Status
Accepted

## Context
Round 1 review flagged both Dockerfiles for shipping devDependencies (drizzle-kit, jest, ts-jest, testcontainers) with no prod prune step. Round 2 review sharpened this: `apps/api/entrypoint.sh` runs `pnpm exec drizzle-kit migrate` at container *runtime*, so `drizzle-kit` is not just image bloat — it's load-bearing in production. A naive `--prod`/prune fix applied to round 1's finding alone would have broken migrations.

## Decision
- Move `drizzle-kit` from `devDependencies` to `dependencies` in `apps/api/package.json`, matching its actual runtime role.
- Add a `pnpm --prod` prune (or equivalent multi-stage deploy step) to both `apps/api/Dockerfile` and `apps/web/Dockerfile`, now safe since the one runtime-required dev tool has been reclassified.

## Consequences
- `jest`, `ts-jest`, `testcontainers`, and other genuine dev-only tools no longer ship in either runtime image.
- `drizzle-kit` ships in the `apps/api` runtime image intentionally — this is expected, not a regression, if a future audit re-flags it.
- Migration behavior (`entrypoint.sh`) is unchanged; no separate init-container/build-step complexity was introduced.
