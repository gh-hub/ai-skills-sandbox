# ADR-001: DB access via NestJS DI module

## Status
Accepted

## Context
`likes.controller.ts` and `app.controller.ts` both import the `db` singleton directly from `db/client.ts` instead of receiving it through NestJS's DI container. This was flagged twice independently (round 1 and round 2 review) as a Shotgun Surgery risk — any change to how the DB connection is constructed or mocked in tests requires touching every import site instead of one provider.

## Decision
Introduce a `DbModule` that:
- Defines a `DATABASE_CONNECTION` injection token.
- Provides the existing `db` client instance (from `db/client.ts`) under that token.
- Is registered `@Global()` so any module in the app can inject it without re-importing `DbModule` everywhere.

`LikesController`/`LikesService` and `AppController` inject `@Inject(DATABASE_CONNECTION) private readonly db: ...` via constructor instead of importing the singleton.

## Consequences
- New modules needing DB access inject the token — no more direct singleton imports to grep for and fix later.
- Test doubles can now override the `DATABASE_CONNECTION` provider in a testing module instead of mocking the `db/client` import.
- `db/client.ts` itself is unchanged — it still constructs the singleton; only its consumption pattern changes.
