# ADR-001: E2E testing architecture

## Status
Accepted

## Context
`apps/web` had no automated test coverage. We needed to choose a testing approach and where it lives in the monorepo.

## Decision
- Tool: Playwright, driving the app through a real browser (Chromium only for v1).
- Target: the real stack — web (nginx/static export), NestJS API, and Postgres — brought up via the existing root `docker-compose.yml`, not mocked network calls.
- Location: a new `apps/e2e` pnpm workspace package, separate from `apps/web`, since the suite is an external consumer orchestrating multiple services.
- Orchestration: Playwright's `webServer` config runs `docker compose up --build` and waits for readiness; there is no separate manual "start the stack" step.
- State handling: tests assert deltas, not absolute values; the DB is reset once per suite run, not per test.
- CI: a new `.github/workflows/e2e.yml` (the repo's first CI workflow) runs the suite on PRs and pushes to `main`.

## Consequences
- Suite runs are slower than a mocked-network approach (full docker-compose build + Postgres startup), but catch real integration bugs between web, API, and DB.
- Adding a new app (`apps/e2e`) means it must be kept in the pnpm workspace and its own dependencies (Playwright) managed independently of `apps/web`.
- Because DB state isn't reset per test, test authors must keep writing delta-based assertions as the suite grows — an absolute-value assertion will be flaky by construction.
- Cross-browser coverage, error/retry UI states, and CI beyond PR/push-to-main triggers are explicitly deferred; revisit if regressions are found in those areas.
